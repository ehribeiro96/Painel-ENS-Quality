# Artifact Storage Policy

## Principles

1. Storage is private by default.
2. The frontend never gets raw filesystem locations.
3. Artifact paths must be generated by the server.
4. Content downloads should be signed and short-lived.
5. Deletion and retention must be auditable.

## Recommended layout

- `data/artifacts/private/metadata/`
- `data/artifacts/private/objects/`
- `data/artifacts/private/tmp/`

## Filename policy

- sanitize display names
- reject path traversal and control characters
- never trust client-supplied storage paths
- store the blob under a content hash, not the original filename

## Access policy

- authenticated management endpoints only
- RBAC for destructive operations
- signed download tokens with HMAC-SHA256
- expiration enforced on every request
- constant-time signature verification

## Future hook

A virus-scan hook should exist before content activation in a later phase.
This phase only reserves the policy; it does not implement scanning.
